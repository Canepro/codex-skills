Checkout API: SLIs, SLOs, and alerting that replace "99.9% uptime"

The checkout API has two different failure modes that a single uptime number cannot see: a synchronous request path where p99 latency drives conversion, and an asynchronous payment-confirmation path customers wait on by email. This plan defines separate SLIs for each, sets objectives with real error budgets, and wires burn-rate alerts sized for a small on-call rotation.

Outcome

Replace one number with four that map to real user harm

Design ready

"99.9% uptime" cannot tell anyone whether checkout is actually healthy: it does not say whether slow requests are killing conversion, and it says nothing about payment-confirmation jobs that customers wait on by email. This plan splits the checkout API into two indicator families that match how it actually fails, sets objectives with a real error budget behind each, and defines burn-rate alerts that a two-to-four person SRE rotation can run without alert fatigue.

Headline takeawayFour SLOs replace the one vague target: request success ratio, p99 checkout latency, payment-confirmation completion ratio, and confirmation delivery time. Each has a budget, an owner, and a burn-rate alert.

Recommended next action

Ship the two request-path SLOs first

Instrument success ratio and p99 latency on the synchronous API this week using existing load-balancer or application metrics. Add the async job SLOs once job-completion events are confirmed to carry timestamps for both enqueue and completion.

Why this orderThe synchronous SLOs need no new instrumentation if requests already carry status code and duration. The async SLOs need a completion event to exist before a burn-rate query is possible, so sequence them second rather than block on both at once.

Constraints

What shaped this design

Context
Two failure modes, one API name

Synchronous checkout requests fail by being slow or erroring in-line. Payment confirmation fails by never completing, or completing too late for the customer to trust the email. A single availability number blends these and hides both.

Small team, no room for noise

A two-to-four person rotation cannot sustain a page for every latency blip. Alerts must be tied to error-budget burn, not to raw threshold breaches, or on-call will start ignoring them within a month.

Business signal already exists

p99 latency is called out because it already correlates with conversion. That makes it a legitimate SLI candidate rather than an internal convenience metric: it maps directly to revenue, which is the strongest justification for an SLO.

Synchronous checkout API

Request success ratio and p99 latency

Primary SLOs

Two SLIs cover the request path: whether a request succeeded, and whether it was fast enough to not cost a sale. Keep them separate. A checkout that returns 200 in 4 seconds is not "available" in any sense that matters to the business, and folding latency into a success ratio hides that.

SLI 1: request success ratio
non-5xx responses ÷ total requests, measured at the load balancer or API gateway
SLI 2: p99 latency
proportion of requests completing under a fixed threshold, not the raw p99 number itself
Threshold
800ms for checkout-submit endpoints (set from current conversion-drop-off data, revisit quarterly)
SLO 1 target
99.9% success ratio over rolling 28 days
SLO 2 target
99% of requests under 800ms over rolling 28 days
Excludes
client-side timeouts before the request reaches the gateway, and known maintenance windows announced 48h ahead
Measurement source
gateway/load-balancer metrics for success ratio (catches upstream outages); application-layer histogram for latency (catches slow dependencies the LB cannot see)

Latency SLI as a threshold, not a raw percentileExpress the SLI as "% of requests under 800ms," not "current p99 value." A raw p99 number cannot be averaged, budgeted, or burn-rated correctly across time windows; a threshold-based ratio can use the same math as an error-rate SLO.

Asynchronous payment-confirmation jobs

Completion ratio and completion-within-deadline

Secondary SLOs

Customers wait on this path by email, so the user-visible promise is not "the job ran," it is "you get a confirmation in a time frame you consider normal." That means two indicators: did the job complete at all, and did it complete before the customer gave up waiting or opened a support ticket.

SLI 3: confirmation completion ratio
jobs reaching a terminal success state ÷ jobs enqueued, measured from job-queue completion events
SLI 4: confirmation delivery time
proportion of jobs completing within a deadline, measured enqueue-to-email-sent
Deadline
5 minutes enqueue-to-email for card payments; longer deadlines for payment methods with inherently slow settlement should get a separate SLO, not be blended in
SLO 3 target
99.5% completion ratio over rolling 28 days (a stuck job is worse than a slow one; it can trigger a support ticket or a duplicate purchase)
SLO 4 target
95% of jobs deliver confirmation within 5 minutes over rolling 28 days
Excludes
jobs that fail due to the payment provider declining the transaction (correct behavior, not a reliability failure); retried jobs count against the deadline from first enqueue, not from the last retry
Measurement source
job-completion events with both enqueue and completion timestamps; do not use worker uptime or queue depth as a proxy, they do not mean the customer got their email

Blind spot to name out loudIf the email provider silently drops messages after your system logs "sent," this SLI reports healthy while customers see nothing. Pair it with a low-volume synthetic check (send-to-self every 15 minutes) or a bounce/delivery webhook if the provider offers one. Do not treat job-completion telemetry alone as proof of customer-visible success.

Error budgets

What each objective allows before it's spent

SLO Target Budget / 28d Owner
Request success ratio 99.9% ~40 min downtime-equivalent Checkout API on-call
p99 latency (under 800ms) 99% ~6.7 hours of slow-request budget Checkout API on-call
Confirmation completion ratio 99.5% ~2 hours of failed-job budget Payments platform on-call
Confirmation delivery time (under 5 min) 95% ~33.6 hours of late-delivery budget Payments platform on-call

Budgets are approximate figures for a rolling 28-day window at each stated target, meant to make trade-offs concrete rather than to be quoted to the decimal place. When a budget is exhausted, the owning team freezes discretionary risk (feature flags, non-critical deploys) on that path until it recovers, per whatever release-gating policy the team adopts.

Alerting

Burn-rate pages, not threshold pages

Alert Condition Route
Fast burn (sync) Burning success or latency budget fast enough to exhaust it in < 2 hours (roughly 14x burn rate over 5m, confirmed over 1h) Page on-call now
Slow burn (sync) Burning budget fast enough to exhaust it in < 3 days (roughly 3x burn rate over 6h, confirmed over 3d) Ticket / next-business-day review
Fast burn (async completion) Completion-ratio budget exhausting in < 2 hours Page payments on-call now
Slow burn (async delivery time) Delivery-time budget exhausting in < 3 days Ticket / weekly SLO review
Stuck-job sweep Any confirmation job older than 15 minutes with no terminal state (catches jobs stuck below the ratio threshold) Page payments on-call now

Why two burn rates per SLOA single fixed threshold either pages too often on brief blips or misses slow erosion entirely. The standard multi-window multi-burn-rate pattern (Google SRE workbook) uses a short, high-multiplier window for acute incidents and a longer, low-multiplier window for sustained erosion, each confirmed against a second window to suppress noise from short spikes.

PromQL sketches

Adapt label names to your actual metric schema. These assume a standard request-duration histogram and a job-completion counter with result and enqueued_at/completed_at semantics captured as a duration histogram.

Sync: request success ratio (fast-burn window) 
1 - (
  sum(rate(checkout_requests_total{code=~"5.."}[5m]))
  /
  sum(rate(checkout_requests_total[5m]))
)
Sync: p99 latency SLI as threshold ratio 
sum(rate(checkout_request_duration_seconds_bucket{le="0.8"}[5m]))
/
sum(rate(checkout_request_duration_seconds_count[5m]))

sum(rate(payment_confirmation_jobs_total{result="success"}[30m]))
/
sum(rate(payment_confirmation_jobs_total[30m]))

/

Fast-burn alert shape (multi-window multi-burn-rate) 
# 14.4x burn rate = exhausts a 28d budget in ~2h if sustained.
# Fire only when BOTH the short and long window agree, to reject blips.
(
  (1 - sync_success_ratio_5m) > (14.4 * 0.001)
  and
  (1 - sync_success_ratio_1h) > (14.4 * 0.001)
)

Adoption

Making this stick with a small team

Operating model
Dashboard

One SLO scorecard per team (checkout API, payments platform) showing current budget remaining and 28-day burn trend. Not a wall of raw metrics.

Review cadence

Weekly 15-minute budget review in the existing team sync, not a separate meeting. Monthly: revisit thresholds (800ms, 5min) against actual conversion and support-ticket data.

Decision hook

Budget exhausted on a path freezes discretionary risk on that path (feature flags, non-urgent deploys) until it recovers. This is the mechanism that makes the SLO matter instead of being a dashboard nobody acts on.

Revision trigger

Revise an SLO when it never burns (too loose, hiding real risk) or burns constantly without matching customer complaints (too strict, wasting on-call attention). Review targets quarterly regardless.

Risks

Open questions before rollout

Risk Severity Owner Mitigation / next step
Job-completion events may lack a reliable enqueue timestamp today Medium Payments platform team Confirm the event schema before committing to SLO 4; add the timestamp field if missing before wiring alerts.
800ms latency threshold is a starting estimate, not yet validated against real conversion data Medium Checkout API owner Pull p99-vs-conversion correlation from analytics before the target is finalized; adjust threshold in week one if data disagrees.
No delivery confirmation from the email provider is assumed in the current design High Payments platform team Add a synthetic send-to-self probe or provider delivery webhook so "job completed" and "customer received email" are not conflated.
Burn-rate multipliers (14.4x / 3x) are the standard SRE-workbook defaults, not yet tuned to this API's traffic pattern Low SRE on-call rotation Run alerts in shadow/silent mode for two weeks and check for false pages before enabling live paging.
